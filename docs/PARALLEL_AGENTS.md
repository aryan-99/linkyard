# Parallel Agents — Operating Model

Linkyard is built by six specialized agents (see `.claude/agents/`) coordinated by an architect. To run them in parallel across multiple terminals without file-conflict chaos, we use **git worktrees**, one per feature per agent.

## Layout

```
/mnt/d/workshop/projects/
├── linkyard/                           ← main worktree, branch: main
│                                          (the architect lives here)
└── linkyard-worktrees/
    ├── backend-auth/                   ← branch: backend/auth
    ├── frontend-search-ui/             ← branch: frontend/search-ui
    ├── extension-save-flow/            ← branch: extension/save-flow
    ├── devops-ci/                      ← branch: devops/ci
    ├── security-dep-audit/             ← branch: security/dep-audit
    └── qa-links-suite/                 ← branch: qa/links-suite
```

Each directory is a real checkout. Terminals work in isolation. Merges happen through the architect via PRs / fast-forwards into `main`.

## Creating a worktree

```bash
cd /mnt/d/workshop/projects/linkyard
./scripts/worktree.sh new backend auth
# → creates ../linkyard-worktrees/backend-auth on branch backend/auth
```

Script commands:

| Command                               | What it does                             |
|---------------------------------------|-------------------------------------------|
| `worktree.sh new <agent> <feature>`   | New worktree + new branch off `main`      |
| `worktree.sh list`                    | Show all worktrees                        |
| `worktree.sh remove <name>`           | Remove a worktree (must be clean)         |
| `worktree.sh prune`                   | Clean up stale refs                       |

Valid agents: `backend frontend extension devops security qa`.

## Starting a Claude session in a worktree

```bash
cd /mnt/d/workshop/projects/linkyard-worktrees/backend-auth
claude
```

Paste this as the **first prompt** (substitute the agent name and feature):

> You are the **backend** agent for Linkyard. Read `.claude/agents/backend.md` and act strictly as that role. Do not edit files outside your scope. Feature: **auth**. Escalate cross-domain questions back to me — I will route them to the architect in the main worktree.

This overrides the "you are the architect" framing in `CLAUDE.md` for that session. The main worktree keeps the architect framing.

## File ownership rules

Some files cut across all surfaces. Only **one** worktree touches them at a time — by default, the **main worktree (architect)**:

- `CLAUDE.md`
- `docs/ARCHITECTURE.md`, `docs/API_SPEC.md`, `docs/DATA_MODEL.md`, `docs/PARALLEL_AGENTS.md`
- `docs/JOURNAL.md` — **always** architect-only; specialists propose entries in their final reports, they do not edit this file
- `README.md`
- `.gitignore`
- `docker-compose.dev.yml` (shared — devops may edit, but coordinate through the architect)
- `.github/` workflows (devops owns, but merges through architect)
- `scripts/worktree.sh`
- Anything at the repo root that isn't inside `backend/ frontend/ extension/`

If a feature *requires* changing a shared file (e.g. new endpoint needs an `API_SPEC.md` update), the agent:
1. Makes the code change in its own worktree.
2. Opens a change request to the architect describing the doc/config update needed.
3. The architect makes the shared-file change in `main`, or batches it into the merge.

This prevents merge conflicts on files nobody thinks to lock.

## Cross-agent communication protocol

Subagents (whether spawned via Task tool or running in parallel terminals) **do not talk to each other directly**. When agent A needs something from agent B:

1. Agent A stops and writes a scoped question: *"Need backend to confirm the shape of `/search` response — `score` float 0–1 or raw distance?"*
2. The question goes to the architect (main worktree).
3. The architect either answers from the docs or routes to agent B.
4. Agent A resumes with the clarified constraint.

This keeps routing centralized and avoids two agents making independent, incompatible decisions about a shared contract.

## Merge workflow

1. Agent finishes the feature in its worktree and runs its **double-check protocol** (from the agent `.md` file).
2. Agent reports to the architect with: files changed, verifications run, open questions.
3. Architect reviews in the main worktree:
   - Checks out or diffs the branch.
   - Runs cross-surface verifications if the change affects contracts (e.g. backend + frontend must agree on API shape).
   - Updates shared files (docs) if needed.
4. Merge to `main` (fast-forward or squash — team preference).
5. Remove the worktree: `./scripts/worktree.sh remove backend-auth`.
6. Other worktrees rebase or merge `main` into their branch to pick up shared changes.

## Conflict hygiene

- **Never** `git push --force` to `main`.
- **Never** edit another agent's worktree from the architect — route the change through the owning agent.
- If two features genuinely need to modify the same file, serialize them through the architect — don't run them in parallel.
- `./scripts/worktree.sh list` should match reality; run `prune` after any manual cleanup.
